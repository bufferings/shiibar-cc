## 背景

Conversations(§4.6)のデータ層。Claude Code の transcript から全会話(live/past フラグ付き)を
SQLite(FTS5 + trigram)に索引し、検索・全文取得の CLI を提供する。消費者はアプリ(M35)と
スクリプトの 2 人で、境界は各コマンドの出力 JSON(§4.4 が公開契約)。**UI はこの指示書の範囲外**。
daemon・プロトコル・hooks・既存コマンドには一切触れない。

### T2: conversations モジュール(transcript と pid ファイルの知識はここだけ)

- 置き場所は **`crates/shiibar-cc` 内の `conversations` モジュール**。次の 4 つを**注入可能**にする:
  transcript ディレクトリ(既定 `~/.claude/projects`)/ pid ファイルディレクトリ(既定
  `~/.claude/sessions`)/ DB パス(既定 = `resolve_state_dir()` の `conversations-index.db`)/
  プロセス生存判定(`kill(0)` + プロセス名)。テストは temp dir のフィクスチャと差し替えた判定だけを
  使う(実ユーザーのデータ・実プロセスに触れない)
- **抽出(1 ファイル)**: 葉(**ファイル内で最後の完全な・sidechain でない user / assistant 行**)から
  `parentUuid` を遡ったアクティブ経路のみを対象に、人の発話(user)と Claude のテキスト返答
  (assistant の text block)を出現順に抽出。親が見つからなければそこを根として打ち切る(§4.6)。
  タイトルは **最後の `custom-title` → 最後の `ai-title` → 最初のユーザープロンプト先頭 → null**(§4.6)。
  cwd も拾う
- **除外**: ツール実行・実行結果・システム行・スラッシュコマンド出力・`<task-notification>` で始まる
  user プロンプト・sidechain(`isSidechain`)。壊れた行・書きかけの末尾行・未知の行 type は skip して続行
- **live/past フラグ**: pid ファイルを列挙し、`kill(pid, 0)` の生存 + プロセス名 = `claude` の
  2 段チェック(§4.6。subprocess を起動しない — sysctl 等のシステムコールで)。
  **pid が 0 以下・非数値のファイルは残骸として扱う**(0 を `kill` に渡すとプロセスグループ全体への
  存在確認になり誤判定するため、渡す前に弾く)
- **§7-6 の「要確認」をここで実物により確定する**: 発話と tool_result 等の弁別 / text block 抽出 /
  タイトル規則(custom-title と ai-title の共存時)/ 葉の決定規則と `leafUuid` の答え合わせ /
  `isSidechain` の意味 / pid ファイルのライフサイクル。確定した内容を完了報告に書く(所有者が §7-6 に反映)

### T3: SQLite 索引と追いつき

- スキーマ(私有 — 外部契約ではない): 会話表(session_id / パス / mtime / サイズ / cwd / タイトル /
  live フラグ)+ 発話表(FTS5 `tokenize='trigram'`。session_id / 出現順 / 役割 / 全文)+
  meta 表(抽出器スキーマ版・構築進捗)
- **SQLite の規律**(§4.6): WAL モード / 書き込みは `BEGIN IMMEDIATE` / **コミットは会話単位** /
  ロック待ち時間は明示設定(5 秒 — 既定に頼らない)/ 全再構築は**ファイル削除ではなく同一接続内の
  テーブル作り直し**(スキーマ版の行を最初のコミットに含め、中断された再構築が次回続きから収束する)
- **追いつき**(`index` / `search` の前段。`show` はやらない — DB 直読。冪等):
  **`<DB パス>.lock` への `flock`** で全体を排他(取得は待つ。プロセス死で OS が解放。DB パスの
  注入に自動で追従 — テストが実 state dir をロックしないこと)→ stat 掃引(mtime はナノ秒精度で
  比較)→ 変更ファイルの全読み直し + 会話単位の入れ替え → 消えたファイルの削除 → フラグ更新。
  状態ディレクトリが無ければ 0700 で作成。DB ファイルは 0600 を明示
- **進捗の中継**(§4.4): 構築の進捗(done/total)を meta 表にコミットごとに記録。ロック待ちに
  なった `index` は、待っている間 meta の進捗を約 250ms ごとに読んで自分の `--json` ストリームとして
  中継する。**待機中の meta 読みは読み取り専用のベストエフォート** — DB 未作成・表なし・旧スキーマ版は
  「進捗なし」として扱い、破損・再構築の判定と実行はロック保持者だけが行う
- スキーマ版不一致・破損検知(open / クエリ失敗)で自動全再構築。手動の再インデックス動詞は作らない

### T4: CLI 動詞(§4.4 の JSON スキーマが公開契約)

- `conversations index [--json]`: 追いつきのみ。`--json` = 進捗の行 JSON(`start` / `progress` /
  `done` / `error`。間引きは §9)、既定は人間向けテキスト(doctor の 2 面性)。処理は新しい順
- `conversations search [<query>] [--json]`: 追いつき後に検索。クエリは**前後の空白(半角・全角)を
  trim し、空白で語に分割して AND**(§4.6 — 全語がタイトル・cwd・本文の合算のどこかに現れる会話)。
  **1 文字の語は無視**(語数の上限は設けない)。有効な語が 1 つも無ければ exit 1 + stderr
  "query too short (minimum 2 characters)"。
  実行は**語ごとに独立のクエリで会話の集合を取り、コードで積集合**する(語の長さによる経路の
  場合分けを作らない)。**クエリは直列で足りる — 並列化はしない**(実測 LIKE 10ms/語・
  MATCH 1ms 未満 — §7-6): **2 文字の語 = LIKE**(`%` `_` エスケープ)/ **3 文字以上の語 = FTS MATCH**
  (**語をフレーズ文字列として引用し、内部の `"` は二重化** — FTS5 のクエリ言語を注入させない)。
  タイトル・cwd もマッチ対象にする方法は自由(スキーマは私有 — 例えば特別な role の FTS 行として
  索引し `show` からは除外する)。会話ごとに 1 件・§4.4 の 5 フィールド。並びは常に新しい順・
  件数上限なし。クエリ無し(trim 後空)は全件。
  **全構築を自分で始める場合は開始時に stderr へ "Building index…" を 1 行**、完了まで待って検索する
- `conversations show <session-id> [--json]`: その会話の発話列**全体**(seq / role / text。text は全文)。
  **追いつきはしない**(DB を読むだけ — §4.6)
- exit code: `index` / `search` = 0(検索 0 件も 0)/ 1。`show` = 0 / 1 / 2(索引に無い)。
  daemon には接続しない
