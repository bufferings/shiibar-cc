## 背景

daemon の停止経路はアプリ quit 時のソケット経由 `{"cmd":"shutdown"}` のみ
(`DaemonLifecycleManager.swift`、DESIGN.md §8.8)。つまり shutdown が届くのは
**いまソケットを握っている daemon 1 プロセスだけ**で、何かの理由でソケットを失った
daemon(アプリのクラッシュ、インストール切り替え中の取りこぼし等)には届く経路がない。

一方 `scripts/dev-uninstall.sh` はアプリ quit のあと `rm -rf $STATE_DIR` でソケット
ファイルごと消す。この時点で生き残っている daemon がいると、到達不能な孤児として確定する
(§4.2 の多重起動防止も、次回アプリ起動時のアタッチ回収も「ソケットファイルが存在する」
ことが前提のため、以後誰からも回収されない)。brew cask 側も同型:
`uninstall quit:` + `zap trash:`(state dir 削除)のみで、孤児には何もしない。

実害の観測: 所有者マシンで、インストール切り替えを挟んだ数日のうちに孤児 daemon が
2 プロセス残留していた(片方はアプリ本体が削除済みのパスのもの)。

ゴール: **dev-uninstall / brew uninstall のどちらでも、ソケットの状態によらず
shiibar-ccd プロセスを残さない。**

## タスク

### T1: dev-uninstall.sh に daemon 直接始末を足す

- `rm -rf $STATE_DIR` の**前**に、プロセス一覧ベースで残存 daemon を終了する 1 ステップを足す:

  ```sh
  # Reap daemons the socket-based shutdown could not reach — a daemon whose
  # socket file was deleted is unreachable by {"cmd":"shutdown"} and is never
  # collected by the duplicate-startup check (DESIGN.md §8.8 / §4.2).
  pkill -xf '/.*/Shiibar CC\.app/Contents/Helpers/shiibar-ccd' || true
  ```

- 置き場所は **app ディレクトリ存在チェックの if の外**(state dir 削除の直前)。
  孤児はアプリ本体がもう存在しなくても残り得るため、無条件に実行する
- `-x` は「コマンドライン全体の一致」を要求するオプション。daemon は引数なしの
  フルパス単体で起動されることを確認済み(`ps` 実測)。パスを**引数に**持つだけの
  無関係プロセス(例: そのパスへの `tail -f` や lldb)を巻き添えにしないための指定
- **macOS の pkill で `-xf` + 正規表現の挙動を実機で一度確認してから確定すること**
  (ダミープロセスを立てて、マッチする/引数に持つだけではマッチしない、の両方向)。
  期待と違う場合は `pgrep -f` で候補を取り、各 pid のコマンドラインを検査してから
  kill する形に落として良い(その場合も理由を完了報告に書く)
- brew 版の現役 daemon もマッチして止まるが、それで良い(dev-uninstall を使うのは
  インストール切り替えの場面。hook 側は daemon 不在を許容する仕様 — §8.8)

### T2: cask テンプレートの uninstall にも daemon 始末を足す

- 対象は `packaging/homebrew/shiibar-cc.rb`(テンプレートが正。tap リポジトリは触らない —
  リリース publish 時に `bump-cask.yml` が反映する)
- 要件: `brew uninstall --cask shiibar-cc` 後に shiibar-ccd プロセスが 1 つも残らないこと。
  既存の `uninstall quit:` は残す(正常系はこれで足りる — 追加分は取りこぼしの保険)
- 実現手段は**現行の Homebrew Cask リファレンスを参照して**選ぶこと(記憶で書かない):
  - `uninstall signal:` が bundle id を持たない裸のヘルパーバイナリ(shiibar-ccd)に
    届くのかをドキュメントで確認する。届かない・不明なら使わない
  - 確実な候補は `uninstall script:` または uninstall 系の flight ブロックで
    T1 と同じ pkill を実行する形。cask 内シェルスクリプトの前例はこのテンプレートの
    `postflight` にある
- 検証: ci.yml の cask check と同じ手順(`{{VERSION}}` / `{{SHA256}}` をダミー値で
  レンダリング → `brew style --cask`)をローカルで通すこと。brew が使えない環境なら
  その旨を完了報告に書き、CI に委ねる

### T3: 検証

- `shellcheck scripts/dev-uninstall.sh` を通す(スクリプト内の既存 `# shellcheck source=`
  ディレクティブと同様、shellcheck 前提で書かれている)
- `cargo test --workspace --locked` / `cargo clippy --all-targets --locked` /
  `(cd app && swift build)` が全緑のまま(いずれも無変更のはずだが確認する)
- 実機スモークは所有者が行う(スモーク項目は指示書の外で共有済み)。
  **brew uninstall の完全な実地検証は次回リリースで cask が更新されるまでできない** —
  代替として、uninstall に足したスクリプト本体を単体で手実行して動作を見せられる形に
