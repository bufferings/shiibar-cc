## 背景とゴール

大量のセッションを開いたとき、一覧が 8 行(360pt)で頭打ちになるのは器の使い方と合わない。
§8.32 の決定: **一時的な器(ドロップダウン)は中身が高さを決め、常駐する器(ウィンドウ)は
使う人が高さを決める**。

## タスク

### T1: ドロップダウンの高さ = 中身(上限は画面)

- 一覧の `maxHeight` 固定値(360)を、**開いた時点のディスプレイの可視領域から計算した上限**に
  置き換える: そのディスプレイの `visibleFrame`(メニューバー・Dock を除く)の高さから、
  topbar・警告行・余白ぶんを引いた値。小さな下端マージン(実機で見て 8〜16pt 程度)を残す
- 計算のタイミングは開いたとき(`captureDropdownOpenTime` の系に乗せるか、パネルの
  `NSWindow.screen` から取る — 最小の確実な方法を選び、理由を報告)。マルチディスプレイでは
  ドロップダウンが実際に出ているディスプレイを使う
- 中身が少ないときは従来どおり中身の高さ(伸ばさない)。ウィンドウ側の一覧には影響させない
  (T2 が別に規定する)

### T2: ウィンドウの縦リサイズ + 高さ記憶

- **Agents ウィンドウの** `windowResizability(.contentSize)` をやめ(Setup Check / Settings の
  同名指定には触らない)、**縦のみリサイズ可**にする(幅は 340 固定 —
  min/max 幅を同値に。最小高はおよそ 3 行ぶん + 信号機の帯)
- 一覧はウィンドウの高さいっぱいに広がり、収まらないぶんはスクロール
  (固定 `maxHeight` を窓では使わない)。中身が少なければ下は空く(自動で縮めない)
- **高さを UserDefaults に記憶**し、次回の Open as Window は「位置 = アイコン直下(従来どおり)、
  高さ = 記憶値」で開く。初回(記憶なし)は一覧の自然な高さ + 帯
- 位置の非記憶・毎回アイコン直下の挙動は変えない(`setFrameTopLeftPoint` の系はそのまま。
  高さ適用と競合しない順序にする)

### T3: 検証

- `(cd app && swift build && swift test)` 全緑。
  `cargo test --workspace --locked` / `cargo clippy --all-targets --locked` 全緑(Rust 無変更)
- view なしで検証できるロジック(上限高の計算式・記憶値の保存/復元の丸め等)があれば
  Core に切り出してテスト(無理に mock を新設しない — 不可能なら理由を報告)
- 実機確認は所有者(スモーク項目: `scripts/dev-demo.sh` で大量セッションを流し込み、
  ドロップダウンが画面いっぱいまで伸びてスクロールに切り替わる / 少数なら従来どおり /
  窓を縦に伸ばす・縮める / 閉じて開き直すと高さが戻る・位置は右上 /
  幅が変えられない / 最小高より縮まない)
