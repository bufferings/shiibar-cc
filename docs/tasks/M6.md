## 背景とゴール

hooks の導入がこれまで「`~/.claude/settings.json` に JSON スニペットを手でマージする」だった。
Claude Code のプラグイン機構(hooks を同梱でき、インストールで自動マージされる)に移行し、
導入を 2 コマンドにする。あわせて uninstall の `--purge` 二段構えを廃止して一段にする。
決定の経緯は DESIGN.md §8.19 / §8.20。

このリポジトリはまだ push されていない(リモートは空)。`/plugin marketplace add bufferings/shiibar-cc`
が動くのは公開後なので、本タスクの検証はローカルパスのマーケットプレイス追加で行う(T6)。

## 事実(検証済み。2026-07-05 に公式ドキュメント code.claude.com/docs/en/plugins-reference.md 等で確認)

- プラグインは `hooks/hooks.json` を同梱でき、hook イベントは settings.json と同じ全種類が使える
- 1 つの GitHub リポジトリがマーケットプレイス(ルートの `.claude-plugin/marketplace.json`)と
  プラグイン本体(相対 source)を兼ねられる
- hooks.json 内からプラグイン同梱ファイルは `${CLAUDE_PLUGIN_ROOT}` で参照する(パスはクォートする)
- インストール(user スコープ)は `~/.claude/settings.json` の `enabledPlugins` に
  `"<plugin>@<marketplace>": true` を書き、プラグイン実体を `~/.claude/plugins/cache/` にコピーする。
  `enabledPlugins` の形は公式ドキュメントに記載あり(安定)。cache の内部構造は非公開(依存しない)
- `claude plugin list --json` という CLI もあるが、doctor は使わない(Node 起動が遅い。
  `enabledPlugins` を読む方式にする。§4.4)

**注意**: `enabledPlugins` のキー・値の実形は、T6 の実機検証で必ず実物を確認してから doctor の
判定を確定すること。ドキュメントの記載と食い違ったら、実装で辻褄を合わせず完了報告で指摘する。

## タスク

### T1: プラグインレイアウトの作成

- `.claude-plugin/marketplace.json`(リポジトリルート): マーケットプレイス名 `shiibar-cc`、
  owner、プラグイン `shiibar-cc`(source `./plugin`)
- `plugin/.claude-plugin/plugin.json`: name `shiibar-cc` / description / version(明示。初期値 `0.1.0` —
  ワークスペース・アプリの `0.1.0` と揃える)
- `plugin/hooks/hooks.json`: 7 イベント(SessionStart / UserPromptSubmit / PostToolUse /
  PostToolUseFailure / Notification / Stop / SessionEnd)をいずれも
  `"${CLAUDE_PLUGIN_ROOT}"/hooks/report.sh <event>` に配線
- `hooks/report.sh` を `plugin/hooks/report.sh` へ移動(`git mv`。実行権限を保つ)。
  スクリプト先頭コメントの `settings-snippet.json` 参照を現行の配布方式の説明に書き換える
- `hooks/settings-snippet.json` と `hooks/` ディレクトリを削除

### T2: install.sh の更新

- `~/.local/bin/report.sh` の設置を廃止(該当 install 行と案内出力を削除)
- 末尾の「hooks setup (manual step)」ブロックを差し替え: スニペット表示・jq マージ案内をやめ、
  `/plugin marketplace add bufferings/shiibar-cc` → `/plugin install shiibar-cc@shiibar-cc` の
  2 コマンドを案内する。導入済み判定(いまは settings.json に `report.sh` があるかで分岐)は
  `enabledPlugins` に `shiibar-cc@shiibar-cc` があるかに置き換える
- ファイル冒頭コメント(「settings.json は触らない」の理由書き)をプラグイン配布前提に書き直す

### T3: uninstall.sh の一段化

- `--purge` フラグと二段分岐を削除。旧 purge の内容(state dir / `cc.shiibar.menubar`
  UserDefaults / 自己署名証明書 / iTerm2 AppleEvents TCC リセット)を常に実行する
- settings.json 編集ブロック(jq フィルタ・バックアップ・手動案内)を削除し、
  hooks の撤去は `/plugin uninstall shiibar-cc` の案内 1 行にする
- `~/.local/bin/report.sh` の削除行は不要になるので消す(install.sh がもう置かないため)
- 通知許可だけは手動(System Settings)という末尾の案内は維持
- ファイル冒頭コメントの二段説明を書き直す

### T4: doctor の hooks チェック書き換え(`crates/shiibar-cc/src/doctor_cmd.rs`)

- 現行: settings.json に文字列 `report.sh` が含まれるか(`hooks_configured`)
- 新仕様(DESIGN.md §4.4): settings.json を JSON としてパースし、`enabledPlugins` オブジェクトに
  キー `shiibar-cc@shiibar-cc` が **true** で存在するか
  - 判定名・表示文言・hint を更新(hint は上記 2 コマンドの案内にする)
  - settings.json が無い / パース不能 / `enabledPlugins` 不在は、いずれも「未導入」として同じ
    warn + hint に畳む。ファイルは在るのに読めない(I/O エラー)は従来どおり別メッセージで
    区別する(現行の分岐粒度を踏襲)
- PATH チェックの hint 文字列 `hooks/report.sh needs it` も現行のパス実態に合わせて更新する
- テスト(同ファイル内 + `tests/exit_codes.rs` に波及があれば追従)を新仕様で書き直す。
  fixtures・テスト内の settings JSON は `enabledPlugins` 形式にする。
  実在の絶対パスを書かない・temp dir を使う(CLAUDE.md のルール)

### T5: README / DEVELOPMENT.md の更新

- README(英語): 「What gets installed」の hooks 項(settings.json のくだり)をプラグイン導入に、
  Install 節の手順(install.sh 実行後にプラグイン 2 コマンド)、Uninstall 節(一段化。`--purge` 削除)、
  「How it works」の `hooks/report.sh` 参照パスを `plugin/hooks/report.sh` に
- DEVELOPMENT.md: settings-snippet 前提の記述(43-44 行付近)と install/uninstall の説明
  (129-137 行付近)を現行仕様に書き直す。ローカルでのプラグイン検証手順(T6 でやったこと)を
  開発メモとして残す
- CLAUDE.md は変更不要のはず。必要を感じたら実装せず完了報告で提案する

### T6: 実機検証(このマシン上。破壊的操作は含まない)

1. `claude plugin marketplace add <リポジトリのローカル絶対パス>` → `claude plugin install
   shiibar-cc@shiibar-cc` を実行(CLI 非対話版。動かなければ `/plugin` 対話版の手順を完了報告に書く)
2. インストール後の `~/.claude/settings.json` の `enabledPlugins` の実形を確認し、
   doctor の判定(T4)と一致していることを確かめる。**実形が仕様と違ったら報告**
3. `shiibar-cc doctor` が hooks チェックを ok にすること、新しい Claude Code セッションを 1 つ
   開いて hook イベントが daemon に届くこと(`shiibar-cc list`)を確認
4. **注意**: このマシンの `~/.claude/settings.json` には旧方式の手動 hooks が入っている。
   **消すのは所有者の作業なので触らないこと**。旧 hooks とプラグインが併存する間、report が
   二重に飛ぶが実害はない(検証はその前提で読むこと)
