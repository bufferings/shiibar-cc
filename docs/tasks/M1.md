# M1 実装指示書: proto + shiibard + `shiibarctl report` + hooks

> 読む順番: ルートの `CLAUDE.md` → この指示書 → 参照節の `docs/DESIGN.md`。
> この指示書は「地図」であり、仕様の正はあくまで DESIGN.md。食い違いを見つけたら実装で解決せず、完了報告で指摘すること。

## ゴール

Claude Code の hook イベントを受けて状態を追跡し、subscribe で配信できる daemon を完成させる。
完了条件(DESIGN.md §6 M1): 遷移表のテーブル駆動テスト + fixtures 再生テストが通り、daemon 再起動で状態が復元されること。

## 成果物

1. **`crates/shiibar-proto`** — メッセージ型(serde)と NDJSON codec
   - ワイヤ形式は DESIGN.md §4.2 のとおり(report / list / subscribe / remove / seen / sessions / info / shutdown、
     snapshot / status_changed / agent_removed、エラー応答 `{"ok":false,"error":…}`)
   - 前方互換: デシリアライズは未知フィールドを無視(serde 既定)。クライアント側 enum は未知値で行ごと失敗させない設計にする
2. **`crates/shiibard`** — daemon 本体
   - 状態機械: DESIGN.md §3.1 の遷移表を**全セル**実装(Notification の種別分岐、SessionStart の source 分岐、
     Stop の background_tasks 分岐、未登録 target 規則、同値遷移の since 維持を含む)
   - エントリ属性と削除規則: §3.2(task / message の上書き・クリア規則、stale 24h、スイープ 60 秒 + 起動時)
   - **Clock を注入可能に**する(stale テストのため)。時刻はすべて daemon の時計(§3.2)
   - subscribe: 接続直後に snapshot を 1 行 push → 以降 status_changed / agent_removed。
     配信条件は status / session_id / cwd / task / message の変化時のみ。遅い subscriber は bounded channel で切断
   - プロトコル契約: §4.2(1 リクエスト 1 レスポンス、report はレスポンスなし・EOF で終了、処理は受信順に直列)
   - 運用: 起動シーケンス(stale socket 処理・二重起動防止)、state.json の atomic 永続化と起動時復元
     (変異のたび。last_seen のみの更新も含む)、sessions.jsonl 追記と重複排除・compaction、
     ログ(stderr、`SHIIBAR_LOG`、report=debug / 遷移と削除=info)、`--foreground` フラグ、shutdown
   - 状態ディレクトリ: 既定 `~/.local/state/shiibar/`(0700)、`SHIIBAR_STATE_DIR` で上書き
3. **`crates/shiibarctl`** — `report` サブコマンドのみ(他のサブコマンドは M2。スタブも作らない)
   - stdin の hook JSON から抽出(§4.1 のフィールド一覧。prompt は先頭 80 文字に切り詰め)、
     target 生成規則(`$ITERM_SESSION_ID` / なければ `session:<session_id>`)、ts 付与(epoch 秒)
   - **常に exit 0**(daemon 不在・タイムアウト 1 秒・不正入力でも。§4.4 の例外規定)
4. **`hooks/report.sh`** と **`hooks/settings-snippet.json`** — §4.1 のとおり
   (7 イベント: SessionStart / UserPromptSubmit / PostToolUse / PostToolUseFailure / Notification / Stop / SessionEnd)
5. **`fixtures/`** — テスト用の hook JSON サンプル。まず公式ドキュメント準拠の手書きサンプルで作る
   (実ペイロードの採取・差し替えは M1 完了後に人間が行う。DEVELOPMENT.md 参照)

## テスト要件(受け入れ条件)

- **遷移表のテーブル駆動テスト**: §3.1 の全セル(イベント種別 × 現状態)を 1 ケースずつ。期待値は表と一対一対応
- **fixtures 再生テスト**: fixtures の hook JSON 列を report 経由で流し、subscribe 出力列(snapshot → 変化イベント)が
  期待と一致することを assert。順序同期は subscribe ストリームのみで行う(sleep 禁止)
- **再起動テスト**: report 数件 → shutdown → 再起動 → list が復元されていること(blocked が残ること)
- **stale テスト**: 注入 Clock を進めて 24h 超過エントリが agent_removed されること
- すべてのテストは `SHIIBAR_STATE_DIR` の一時ディレクトリで動くこと(実ディレクトリに触れたら不合格)
- `cargo test` 全緑、`cargo clippy --all-targets` 警告ゼロ

## スコープ外(やらない)
