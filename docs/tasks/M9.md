## 背景とゴール

Mission Control の「アプリケーションの切り替え時に、アプリケーションのウインドウがあるスペースに移動」
が OFF のマシンでは、focus の select + activate がウィンドウのある Space / ディスプレイに
切り替わらず、「Setup Check 全緑なのに飛ばない」という診断の盲点になる(2026-07-07 に 2 台目の
実機で確認。DESIGN.md §7-1)。doctor にこの設定のチェックを 1 項目追加し、盲点を塞ぐ。

## 事実(実機検証済み)

- 設定の実体は **NSGlobalDomain の `AppleSpacesSwitchOnActivate`**
- `defaults read -g AppleSpacesSwitchOnActivate` は、明示的に OFF にすると `0`、ON にすると `1` を
  返し、一度も触っていないマシンではキー不在(exit 非 0)のことがある。既定は ON
- OFF のマシンで ON に切り替えると focus が正常化することを確認済み(因果確定)

## タスク

### T1: doctor のチェック追加(`crates/shiibar-cc/src/doctor_cmd.rs`)

- 新チェック `spaces`(id は実装で一貫していれば別名でも可)を TCC チェックの後に追加:
  - 値が明示 `0` → **warn**。summary は設定が OFF であること、hint は
    System Settings > Desktop & Dock の "When switching to an application, switch to a Space
    with open windows for the application" を ON にする案内(英語)
  - `1`・キー不在・読み取り失敗 → **ok**(§4.4: 既定が ON のため、誤警告より見逃しを選ぶ)
- 値の取得は `defaults read -g AppleSpacesSwitchOnActivate` のサブプロセス実行でよい。
  ただし doctor の設計原則(チェック対象は明示パラメータで注入し、実環境なしで単体テスト可能)を守ること —
  既存の `AppleScriptRunner` と同様に、読み取りを抽象(クロージャ or trait)にして注入する
- 新しい外部クレートを追加しない

### T2: テスト

- 単体テスト: `0` → warn(hint 付き)/ `1` → ok / 読み取り失敗(コマンド不発・非 0 exit・空出力)→ ok
- 既存の doctor テスト(全チェック列の JSON 形など)への波及を追従
- 実環境の `defaults` に依存しない(注入した偽の読み取りで完結させる)

### T3: 検証

- 実機での `shiibar-cc doctor` 実行確認は所有者側で行う(このマシンは設定 ON なので ok 表示の確認のみ)
