## 背景とゴール

hooks を取りこぼすと status がドリフトする(実観測: 「working のまま固まる」)。既存の修復契機は
起動時・daemon 再接続時・手動 Rescan のみで、放置中のドリフトは直らない。§8.10 で予約していた
定期 reconcile を有効化する(経緯と設計判断は DESIGN.md §8.22)。

## タスク

### T1: 定期 reconcile(`app/Sources/ShiibarCcApp/AppState.swift` ほか)

- `NSBackgroundActivityScheduler` で約 60 秒間隔(`interval = 60`、`tolerance = 30`、
  `repeats = true`、`qualityOfService = .utility`)の定期実行を追加し、既存の
  `runReconcile(showFeedback: false)` を呼ぶ(§4.5: 定期実行はフィードバック表示なし —
  既存実装がすでに手動 Rescan だけにフィードバックを限定している)
- スケジューラの起動はアプリ起動時に 1 回。停止はアプリ終了に任せてよい(daemon と違い
  明示的なライフサイクル管理は不要)が、AppState の deinit 等で invalidate するのが自然なら行う
- 間隔・tolerance の値は §9 の定数として 1 箇所に定義する(既存の定数の置き方に合わせる)
- **自前の画面状態・スリープ検知は書かない**(§8.22: OS のスケジューラに委ねる)
- reconcile は冪等(既存コメントに明記あり)なので、手動 Rescan・起動時実行との重なりは
  考慮不要。in-flight の直列化などの新規機構を足さないこと

### T2: テスト

- スケジューラ自体は OS の機能なので、テストは切り出せるロジックに限ってよい
  (例: 定数値、スケジューラ設定を組み立てるヘルパーがあればその値の検証)。
  無理に発火タイミングをテストしない
- 既存の Swift テスト(88 件)が緑のままであること

### T3: 検証

- `cargo test --workspace --locked` / `cargo clippy --all-targets --locked` 全緑・警告ゼロ
  (Rust 側に変更はないはずだが確認)
- `(cd app && swift build && swift test)` 全緑
- 実機での長時間観察(ドリフトが自然修復されること)は所有者側で行う
