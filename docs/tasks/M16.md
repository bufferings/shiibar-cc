## 背景とゴール

所有者の実機観察: Setup Check ウィンドウを閉じて開き直すと、チェックが再実行されず
前回の結果が古いまま表示される。仕様(DESIGN.md §4.5)は「チェックはウィンドウを
開くたびに自動実行する(閉じて開き直した場合も含む)」。

現実装は `SetupCheckView` の `onAppear` で `runChecks()` を呼んでいるが、
ウィンドウは `Window` シーン + `openWindow(id:)`(`ShiibarCcMenuBarApp.swift`)で
開かれており、閉じて開き直したときに `onAppear` が再発火しない経路があるとみられる。

## タスク

### T1: 開き直しでも確実に自動実行させる

- まず現象を実装レベルで特定する(`onAppear` がいつ発火し、いつしないか。
  観察をコメントか完了報告に残す)
- そのうえで、**開くたびに `runChecks()` が走る**最小の仕組みに直す。候補(どれを選ぶかは
  発火条件の観察結果に従う。凝った仕組みにしない):
  - ウィンドウの key 化/表示の通知(`NSWindow.didBecomeKeyNotification` 等)を自ウィンドウに
    限定して受け、閉→開をまたいだ再表示で実行する
  - あるいは開く側(`DropdownView` の Setup Check メニューハンドラ)から実行をトリガーする
- 制約:
  - 実行中の多重実行ガード(`runChecks()` の in-flight guard)は維持。開き直しの連打で
    チェックが並走しないこと
  - ウィンドウが開いたまま前面/背面を行き来しただけで再実行が走り続けない形が望ましい
    (走っても実害は小さいが、素朴に didBecomeKey だけで発火させると Re-run 中の表示と
    競合しないか確認すること)
  - チェックの実行と判定ロジック(`SetupCheckLogic` / `SetupCheckParsing` / `CLIRunner`)には
    触らない。`NSApp.activate` は維持
  - Settings ウィンドウには手を入れない(開くたびに実行するものが無い)
