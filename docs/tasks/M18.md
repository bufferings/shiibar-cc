## 背景とゴール

v0.1.0 配布マイルストーンの後半。M17 のパイプライン(タグ → 公証済み zip の draft Release)を土台に、正式導入経路を作る:

```
brew install --cask bufferings/tap/shiibar-cc
```

この 1 コマンドで .app が `/Applications` に入り、CLI 2 本が brew の bin に symlink され(hooks の `plugin/hooks/report.sh` は PATH の `shiibar-cc` に依存 — これで満たされる)、**postflight が Claude Code の hooks plugin を自動インストールする**(初回ガード付き。下記)。cask 定義の正はこのリポジトリに置き、リリース publish 時に workflow が tap リポジトリ `bufferings/homebrew-tap` へ生成コミットを push する。

## 決定済みの設計(逸脱する場合は報告)

- 正式経路 = brew cask(自前 tap)。arm64 のみ。**Intel Mac はサポートしない**(動作を確認する実機も CI もないため、約束しない)。ソースビルドは開発者向けの経路で、Intel 対応の代替としては扱わない。
- plugin の自動インストールは**初回ガード付きの postflight**。自動で入ることを README と caveats の**両方に明記**する(黙って入れない — 所有者の明示要求)。
- draft Release の publish(所有者スモーク後)→ `release: published` で cask を bump。cask が未公開・未スモークの資産を指すことは構造上あり得ない形にする。
- plugin はリリース zip ではなく**リポジトリ main から**入る(`claude plugin install` の仕様)。バイナリと hooks の版ずれは、report.sh が「shiibar-cc 不在なら黙って exit 0」の薄い転送層であることで許容する。

## タスク

### T1: cask テンプレート `packaging/homebrew/shiibar-cc.rb`(新規)

プレースホルダ `{{VERSION}}` / `{{SHA256}}` を持つテンプレート。骨格:

- `version` / `sha256` / `url "https://github.com/bufferings/shiibar-cc/releases/download/v#{version}/shiibar-cc-#{version}-arm64.zip"`(M17 の資産名と一致させる)/ `name "Shiibar CC"` / `desc` / `homepage`
- `depends_on macos: ">= :ventura"`(= 13)/ `depends_on arch: :arm64`
- `app "Shiibar CC.app"`
- `binary "#{appdir}/Shiibar CC.app/Contents/Helpers/shiibar-cc"` と同 `shiibar-ccd`
- `postflight`: 下記の設計
- `uninstall quit: "cc.shiibar.menubar"`
- `zap trash:`: `~/.local/state/shiibar-cc`(state dir)/ `~/Library/Preferences/cc.shiibar.menubar.plist` / `~/Library/Saved Application State/cc.shiibar.menubar.savedState`。
  **`~/.claude/` 配下と `~/.local/bin` には絶対に触らない**(前者は Claude Code の管轄 — plugin の撤去は caveats で `claude plugin uninstall shiibar-cc` を案内する。後者はソースビルド経路の持ち物)
- `caveats`: (1) postflight が hooks plugin の自動インストールを**試みた**ことの開示と、手動フォールバックの 2 行(`claude plugin marketplace add bufferings/shiibar-cc` / `claude plugin install shiibar-cc@shiibar-cc`)、(2) アプリを一度開いて通知・iTerm2 Automation の許可を出すこと(Login Item 登録は初回起動で自動)、(3) `shiibar-cc doctor` での確認、(4) plugin の撤去コマンド、(5) 通知許可の削除だけはシステム設定でしかできないこと

**postflight の設計**(shell に落とす。`must_succeed: false` 相当 + 全コマンド `|| true` で、**どんな失敗でも cask のインストールは壊さない**):

1. claude の解決は利用者の対話シェルの PATH に頼らない: `command -v claude` → 無ければ `$HOME/.local/bin/claude` を直接確認。どちらも無ければ何もせず終了(caveats の手動 2 行が受け皿)
2. **初回ガード**: `$HOME/.claude/settings.json` の `enabledPlugins` に `"shiibar-cc@shiibar-cc"` のエントリが**値を問わず**(true でも false でも)既にあれば何もしない — `brew upgrade` での再実行や、利用者が意図的に無効化・撤去した plugin を勝手に復活させないため。判定は jq に依存しない(grep で足りる。`scripts/install.sh` の既存チェックが同系)
3. ガードを通ったら `claude plugin marketplace add bufferings/shiibar-cc` → `claude plugin install shiibar-cc@shiibar-cc`

### T2: `.github/workflows/bump-cask.yml`(新規)

- トリガー: `release: types: [published]`。`if: ${{ !github.event.release.prerelease }}`。runner は `ubuntu-latest`
- ステップ: このリポジトリを **release のタグで** checkout(テンプレートをリリース時点の内容で使う)→ `gh release download <tag> -p '*-arm64.zip'` して `sha256sum`(**公開済み資産から再計算** — ビルドログの値を信用しない)→ `bufferings/homebrew-tap` を `secrets.TAP_PUSH_TOKEN` で checkout → テンプレートの `{{VERSION}}` / `{{SHA256}}` を置換して `Casks/shiibar-cc.rb` に配置 → `shiibar-cc <version>` のメッセージでコミット・push
- secret 名(確定): `TAP_PUSH_TOKEN`

### T3: ci.yml への cask 検査の追記

macOS ジョブに 1 ステップ: テンプレートをダミー値(`0.0.0` / ダミー sha256)で描画し、`brew style --cask <描画結果>` を通す(テンプレートが Ruby として妥当なことが普段の CI で腐らないように。数秒で済む検査なので dry-run 送りにしない)。既存ステップは変えない。

### T4: ローカル用スクリプトの改名(git mv)

brew が正式経路になるので、ローカルスクリプトは「開発用」であることが名前で分かる形に揃える:

- `scripts/install.sh` → `scripts/dev-install.sh`
- `scripts/uninstall.sh` → `scripts/dev-uninstall.sh`
- `scripts/dev-reload.sh` はそのまま(既に用途が名前になっている)

改名は `git mv` で行い、**リポジトリ内の参照を全て更新する**(スクリプト自身のヘッダコメント・相互参照、`docs/DEVELOPMENT.md`、`README.md`、`docs/DESIGN.md` に旧名があればそれも)。`SHIIBAR_CC_BIN_DIR` / `SHIIBAR_CC_APP_DIR` などの環境変数名は変えない。

### T5: ドキュメント更新

- `README.md`:
  - Install を書き換え: 正式経路 = `brew install --cask bufferings/tap/shiibar-cc`(要件: macOS 13+ / **Apple Silicon** / Homebrew)。**postflight が hooks plugin を自動インストールすることを明記**し、手動フォールバックの 2 行も載せる。導線は「brew install → アプリを開いて許可 → Setup Check(または `shiibar-cc doctor`)で確認」
  - clone + `./scripts/dev-install.sh` は「ソースビルド(開発者向け)」の節として残す。Intel Mac 向けの経路としては書かない(サポートしない)
  - セッション内 `/plugin ...` 表記をシェルの `claude plugin ...` に統一(uninstall 含む)
  - Uninstall: brew 経路は `brew uninstall --cask shiibar-cc`(完全削除は `--zap`)+ `claude plugin uninstall shiibar-cc`。通知許可の手動削除の記述は維持
  - Permissions 節の自己署名証明書の記述を「ソースビルド経路のみ」に限定(brew 版は Developer ID 署名・公証済み)
- `docs/DEVELOPMENT.md` の「リリース・インストール」節に追記:
  - リリース手順: タグ push(タグ = Cargo.toml の version と一致必須)→ Actions が draft Release を作る → 所有者が draft の zip を実機スモーク → publish → bump-cask.yml が tap を更新 → **publish 直後に Cargo.toml の version を次の番号(まず patch)へ bump してコミットする**。**公開後のタグは再利用禁止**(公開前に失敗した draft はタグごと消してよい)
  - バージョンの原則も明文化する: リポジトリの Cargo.toml の version は常に「次に出す番号」。dev ビルドはそれに `-dev` を付けて名乗る(About で見分けられる)。機能が入るリリースは 0.x を、修正だけなら 0.x.y を上げる(publish 後はまず patch に bump し、リリース時に実態へ合わせてよい)
  - 使う secrets の一覧(M17 の 5 つ + `TAP_PUSH_TOKEN`)。値の作り方は書かない(名前と用途だけ)
  - 切り替え手順(dev 版と brew 版は同居させず、常にどちらか一方): リリース検証・移行は `scripts/dev-uninstall.sh` → `brew install --cask ...`、開発に戻るときは `brew uninstall --cask shiibar-cc` → `./scripts/dev-install.sh`。署名の切り替え(自己署名 ↔ Developer ID)で通知と Automation の許可の再確認が毎回 1 回ずつ出るのは想定内(切り替えはリリース検証時だけなので頻度は低い)
- `docs/DESIGN.md`: §4.5 の「配布はしない(当面自分専用)」とその周辺(ad-hoc/ローカル署名前提の記述)を、配布ありの現状(brew cask = Developer ID 公証済み zip / ソースビルド = ローカル署名)に合わせて更新する。**§8(決定ログ)には一切触らない** — §8 への追記は別途所有者承認のもとで行われる。
