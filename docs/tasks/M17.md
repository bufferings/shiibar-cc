## 背景とゴール

v0.1.0 配布マイルストーンの前半。正式導入経路を brew cask にする(cask・tap・README は次の M18)。
その土台として、このマイルストーンで作るのは:

1. **.app 組み立ての単一ソース化**: いま `.app` の組み立て(バンドル構成・Info.plist 生成・アイコン・署名の重ね順)は `scripts/install.sh` にしかなく、署名部分は `scripts/dev-reload.sh` にも重複している。これを `scripts/lib/bundle.sh` に抽出し、ローカルインストール・dev-reload・リリースビルドの 3 者が同じ実装を使う形にする。
2. **リリースパイプライン**: タグ `vX.Y.Z` の push で GitHub Actions が arm64 ビルド → Developer ID 署名(hardened runtime)→ 公証 → staple → zip → **draft** の GitHub Release 作成(リリースノート自動生成付き)まで行う。draft の publish は所有者の実機スモーク後の手動操作(このリポジトリの原則「push = 公開の前に所有者スモーク」をタグ配布にも適用するため。draft は所有者にしか見えない)。

このマイルストーンでは**ドキュメント(README / docs/DEVELOPMENT.md / docs/DESIGN.md)には一切触らない**。それは M18 と、その後の別作業で行う。

## 前提となる検証済みの外部事実(再調査不要。逸脱する場合は報告)

- `xcrun notarytool submit` は App Store Connect API キー(`--key <p8 のパス>` / `--key-id` / `--issuer`)と `--wait` に対応し、zip を提出できる(ローカルの Xcode 16.2 の実物 help で確認済み)。
- 公証には Developer ID Application 署名 + **hardened runtime**(`--options runtime`)+ secure timestamp(`--timestamp`)が必須。**バンドル内の全実行ファイルが対象**(アプリ本体だけでなく `Contents/Helpers/` の 2 バイナリも)。無いと "The executable does not have the hardened runtime enabled" で Invalid になる。
- staple は **.app に対して**行う(`xcrun stapler staple`)。公証に提出した zip はそのまま配布物にできない — **staple 後に zip を作り直す**。
- GitHub Actions への証明書 import は GitHub 公式手順: base64 の .p12 を復号 → `$RUNNER_TEMP` 配下の一時キーチェーンに `security import` → `security set-key-partition-list -S apple-tool:,apple:`。**Developer ID 配布に provisioning profile は不要**。
- entitlements は付けない(time-sensitive は見送り済み。Apple Events 用 entitlement は公証ビルドの実機スモークで focus が壊れた場合に後続で足す方針 — このマイルストーンでは実装しない)。

## タスク

### T1: `scripts/lib/bundle.sh` の抽出と 3 スクリプトの改修

`scripts/lib/signing.sh` と同じ「source して使う関数ライブラリ」形式で `scripts/lib/bundle.sh` を新設する。

- `assemble_app_bundle <app_path> <app_bin> <ccd_bin> <cc_bin> <version> <bundle_version>`:
  現在の install.sh の組み立て部分(rm -rf → `Contents/MacOS` + `Contents/Helpers` 作成 → 3 バイナリの `install -m 755` → Info.plist の heredoc 生成 → `scripts/generate-app-icon.swift` + `iconutil` でアイコン生成・設置)を移す。
  Info.plist は `CFBundleShortVersionString = <version>`、`CFBundleVersion = <bundle_version>` をパラメータ化する。それ以外(bundle id `cc.shiibar.menubar`、`LSUIElement`、`LSMinimumSystemVersion 13.0`、copyright)は現状維持。
- `sign_app_bundle <identity> <app_path> [extra codesign flags...]`:
  現在の重ね順(Helpers の `shiibar-ccd` → `shiibar-cc` を個別に署名 → 最後にバンドルを `--identifier cc.shiibar.menubar` 付きで署名)を維持し、追加フラグ(`--options runtime --timestamp` 等)は **3 回の codesign 全部に**適用する(上の「全実行ファイルが対象」のため)。

追加(実機スモークでの所有者指示 2026-07-08): dev-reload.sh の「アプリ quit → daemon を確実に止める」ブロックを `scripts/lib/app-lifecycle.sh`(`wait_for_gone` / `stop_app_and_daemon`)として抽出し、**install.sh も組み立ての前に呼ぶ**。理由: 起動中のアプリへの再インストールで、末尾の `open` は既存プロセスを前面に出すだけで再起動しないため、旧プロセス・旧 daemon が残り続ける。

改修する 3 スクリプト:

- `scripts/install.sh`: 組み立て・署名を bundle.sh の関数呼び出しに置き換える。
  - `CFBundleShortVersionString` のハードコード `0.1.0` を廃止し、ルート `Cargo.toml` の `[workspace.package]` の `version`(現在 10–11 行目)から読み、**`-dev` サフィックスを付けて**渡す(例 `0.1.0-dev`)。ローカルビルドとリリースビルドを About パネルで見分けるため(リリースビルドは build-release-app.sh 経由で素の番号を渡す)。
  - `CFBundleVersion` はローカル用のタイムスタンプ(`BUILD_STAMP`)を**維持**する(キャッシュ破りの理由コメントも維持)。
  - restricted entitlement の説明コメント、symlink・PATH 警告・enabledPlugins チェック・lsregister・`open` の流れは現状維持。
  - 末尾の hooks 案内 2 行を、セッション内 `/plugin ...` からシェルコマンドに変える:
    `claude plugin marketplace add bufferings/shiibar-cc` / `claude plugin install shiibar-cc@shiibar-cc`
- `scripts/dev-reload.sh`: インラインの `sign_all` を `sign_app_bundle` 呼び出しに置き換える。**バイナリ差し替え方式(全体を組み立て直さない)は維持**。既存の説明コメントは失わない。
- `scripts/uninstall.sh`: plugin 撤去の案内を `claude plugin uninstall shiibar-cc` に更新する(`uninstall|remove` サブコマンドは現行 claude CLI の実物 help で存在確認済み)。

### T2: リリース用スクリプト(`scripts/release/` 新設)

いずれも bash、`set -euo pipefail`、機械固有パスなし、CI とローカルの両方で動くこと。

- `check-version.sh <tag>`: `vX.Y.Z` の `v` を外し、ルート `Cargo.toml` の `[workspace.package]` の `version` と比較。不一致なら両方の値を出して非ゼロ終了。
- `build-release-app.sh <version> <out_dir>`:
  1. `cargo build --release --locked -p shiibar-ccd -p shiibar-cc`
  2. `(cd app && swift build -c release)`、バイナリパスは `swift build -c release --show-bin-path`
  3. `assemble_app_bundle "<out_dir>/Shiibar CC.app" ... <version> <version>`(**リリースの CFBundleVersion = version**。タイムスタンプにしない — スモークした物と公開する物を同一に保つため)
  4. 署名 identity の解決: 環境変数 `SHIIBAR_CC_SIGN_IDENTITY` があればそれを使う(ad-hoc `-` を含む。CI ガードやローカル試験用)。無ければ `security find-identity -v -p codesigning` から `Developer ID Application` を探す。見つからなければ非ゼロ終了。
  5. `sign_app_bundle <identity> <app> --options runtime --timestamp`(ad-hoc のときは `--timestamp` を外してよい — ad-hoc はタイムスタンプサーバーを使えないため。分岐する場合はコメントで理由を書く)
  6. `codesign --verify --deep --strict` を通す。
- `generate-release-notes.sh <tag>`: 前タグ(`git describe --tags --abbrev=0 "<tag>^"`)〜 `<tag>` のコミット件名を `- <subject>` 形式で stdout に出す。`Close the M<数字>` で始まる件名は除外。**前タグが無い場合(v0.1.0 がまさにこれ)は全履歴を対象にする**。

### T3: workflows

- `.github/workflows/release.yml`(新規): トリガー `push: tags: ["v[0-9]+.[0-9]+.[0-9]+"]`、`permissions: contents: write`、runner は `macos-15`(ci.yml と同じ。arm64)。ステップ順:
  1. checkout(`fetch-depth: 0` — リリースノートが全履歴を要るため)
  2. `check-version.sh "$GITHUB_REF_NAME"`
  3. 証明書 import(上記の公式手順。キーチェーンパスワードは実行ごとに `uuidgen` で生成し secret にしない)
  4. `build-release-app.sh <version> dist`
  5. 公証: `APPLE_API_KEY_P8` を `$RUNNER_TEMP/AuthKey.p8` に書き出し → `ditto -c -k --keepParent "dist/Shiibar CC.app" dist/notarize.zip` → `xcrun notarytool submit dist/notarize.zip --key ... --key-id ... --issuer ... --wait`。失敗時は `xcrun notarytool log <submission-id>` をステップ出力に出す
  6. `xcrun stapler staple` → `xcrun stapler validate` → `spctl -a -vv "dist/Shiibar CC.app"`
  7. **staple 後に** `ditto -c -k --keepParent` で `dist/shiibar-cc-<version>-arm64.zip` を作る(notarize.zip は使い回さない)。`shasum -a 256` をログに出す
  8. `generate-release-notes.sh "$GITHUB_REF_NAME"` → `gh release create "$GITHUB_REF_NAME" --draft --verify-tag --title "$GITHUB_REF_NAME" --notes-file ... <zip>`
  9. 後始末(一時キーチェーン削除・AuthKey.p8 削除)を `if: always()` で
  - 使う secrets(名前はこれで確定): `DEVELOPER_ID_CERT_P12`(base64 .p12)/ `DEVELOPER_ID_CERT_PASSWORD` / `APPLE_API_KEY_P8`(.p8 の中身)/ `APPLE_API_KEY_ID` / `APPLE_API_ISSUER_ID`
  - **dry-run**: トリガーに `workflow_dispatch` も足す。手動起動時はタグ push と同じステップを本物の secrets で最後まで走らせ(証明書 import・公証・staple 含む)、**draft Release の作成だけスキップ**して zip を workflow artifact としてアップロードする。バージョンはタグが無いので Cargo.toml の `[workspace.package]` の version を使い、check-version.sh はスキップする。用途は本番タグの前に secrets・パイプライン一式をリハーサルすること。分岐は最小限に(バージョンの取得と最終ステップの 2 か所)
- `.github/workflows/ci.yml` には触らない。
